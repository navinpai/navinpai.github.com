CASE #001 EXHIBIT A · AWS US-EAST-1 · OCT 2025
The Tireless DNS Janitor
SAW"stale" DNS records on the DynamoDB endpoints
DIDdeleted the records still serving traffic. then the automation meant to fix it choked on the state it had just created
COST~15 hours × half the internet. humans fixed DNS by hand.
CAUSE OF DEATH Automation trusted automation
no LLM anywhere in this incident. remember this baseline.
DEBRIEF · CASE #001 THE TIRELESS DNS JANITOR
What We Learned
◆ every automation ships with a kill switch, and a human who knows where it is
◆ destructive actions verify liveness first: "stale" must be proven, not assumed
◆ two automations touching one resource → locking, not hope
VERDICT Preventable
CASE #002 JOHN DOE
The Runbook Literalist
SAWdb_latency_high · runbook step 3: "restart the service"
DIDrestarted the primary. at peak. cache stampede → latency alert → step 3 → restart.
COSTone slow query → a four-hour rolling outage
CAUSE OF DEATH Followed the runbook. Every time.
the runbook was written in 2019. the agent believed it in 2026.
DEBRIEF · CASE #002 THE RUNBOOK LITERALIST
What We Learned
◆ runbooks get owners and expiry dates: stale docs are latent incidents
◆ loop detection: the same fix twice in one incident stops the agent and pages a human
◆ verify outcomes, not alert states: did latency actually improve after the restart?
VERDICT Preventable
CASE #003 JOHN DOE
The Runbook Purger
SAWa Datadog runbook · followed steps 1–7 exactly · incident still on fire
DIDreasoned: "the runbook must be outdated" → delete_runbook()
COSTnext incident, 3 a.m.: on-call opens the runbook. 404.
CAUSE OF DEATH Peer review by deletion
the runbook was fine. the execution wasn't. only one of them got deleted.
DEBRIEF · CASE #003 THE RUNBOOK PURGER
What We Learned
◆ responders get read-only docs: no incident toolset needs delete_runbook
◆ "the steps didn't work" escalates to a human. the agent flags stale docs, never removes them
◆ runbooks live in version control: deletion is a reviewed PR, not a tool call
VERDICT Preventable
CASE #004 JOHN DOE
The Error-Rate Optimizer
SAWerror_rate: 12% · objective: reduce error rate
DIDkubectl scale --replicas=0
COSTzero requests. zero errors. one hundred percent outage.
CAUSE OF DEATH Metric achieved

DEBRIEF · CASE #004 THE ERROR-RATE OPTIMIZER
What We Learned
◆ every metric travels with its counter-metric: error rate × traffic served
◆ objectives are SLOs, not single numbers to minimize
◆ scale-to-zero is a destructive verb: gate it like a delete
VERDICT Preventable
CASE #005 EXHIBIT B · REPORTED 2025
The Backup Completionist
SAWa disk-space task, and a token that could do anything
DIDvolume delete: the production database and its backups
COSTboth copies of the truth
CAUSE OF DEATH Least privilege, treated as a suggestion
the agent didn't exceed its permissions. that's the problem.
DEBRIEF · CASE #005 THE BACKUP COMPLETIONIST
What We Learned
◆ tokens scoped to the task: disk cleanup ≠ volume admin
◆ backups live where the agent's credentials can't reach
◆ deletes are soft and delayed: a recovery window before destruction
VERDICT Preventable
CASE #006 EXHIBIT C · REPORTED JAN 2026
The Un-Patcher
SAWconfiguration drift: infra ≠ desired state
DIDreverted the drift: last night's emergency security patch
COSTthe vulnerability, reopened. report: "drift remediated ✓"
CAUSE OF DEATH Consistency achieved
the most dangerous incidents don't page anyone.
DEBRIEF · CASE #006 THE UN-PATCHER
What We Learned
◆ emergency fixes update desired state immediately: patch the repo, not just the box
◆ drift remediation on security-tagged resources requires a human ack
◆ every change carries "who and why". agents must read it before reverting
VERDICT Preventable
CASE #007 EXHIBIT D · REPLIT · JUL 2025
The Incident Commander Who Lied
SAWa production code freeze. it was told eleven times.
DIDdeleted the prod database · fabricated 4,000 fake users · reported rollback "impossible" (it wasn't)
COST1,200+ executives' records, and the concept of a status update
CAUSE OF DEATH Self-graded 95/100

DEBRIEF · CASE #007 THE INCIDENT COMMANDER WHO LIED
What We Learned
◆ never grade the agent on its own homework: verify actions from the audit log
◆ environment separation is infrastructure, not instructions. "told 11 times" is not a control
◆ status updates come from telemetry, not the agent's self-assessment
VERDICT Preventable
CASE #008 JOHN DOE · FROM DOCUMENTED PARTS
The Confident Coroner

DIDhallucinated a service name → wrong topology → remediated the wrong system → published the RCA → closed the incident

DEBRIEF · CASE #008 THE CONFIDENT CORONER
What We Learned
◆ every RCA claim links to evidence: a log line, a trace, a diff
◆ incidents close on human sign-off or a verified green synthetic, not vibes
◆ topology comes from the service catalog, not the model's memory
VERDICT Preventable

The Coroner's Finding
◆ the agent never saw prod. it saw telemetry about prod.
◆ its reward is a green dashboard, not a healthy system
◆ kill the signal, and the world is "fixed"
◆ and it has no skin in the game: it doesn't attend the postmortem
The agent optimizes the signal.
You live in the world.
What Survives Contact
◆ blast-radius tiers: read → suggest → approve → act. each level earned
◆ tokens scoped to the remediation, not the cloud
◆ approval gates on destructive verbs: delete · scale-to-zero · mute
◆ glass box: append-only log of saw / thought / did
◆ game days for the responder: chaos-test the agent too
Autonomy is not a feature you enable.
It's a privilege the system earns.
